# 文档维护规范

用户手册的目标是帮助科研用户完成可重复工作流，而不是展示所有实现细节。

## 基本规则

- 用户手册放在 [ECAT 仓库 `docs/`](https://github.com/kefuhe/ECAT/tree/main/docs)。
- 不在公开手册中引用本地绝对路径、未整理笔记或未随手册发布的材料。
- 优先写经过整理的工作流，不直接堆脚本。
- 每个案例都要有可复现命令、输入说明和输出说明。
- 每个案例的教学重点都要落到脚本或配置片段上，说明“这段代码做什么、为什么要这样写、下一步接到哪里”。
- 大型 HDF5 样本和图件应放在 [ECAT-Cases](https://github.com/kefuhe/ECAT-Cases)，不要放进 [ECAT](https://github.com/kefuhe/ECAT) 主代码仓库。
- 导航入口使用“中文 / English (`directory/`)”格式，例如“工作流 / Workflows (`workflows/`)”，让用户能直接对应 GitHub 目录。
- 跨页链接如果跳到小节，优先使用稳定英文锚点；中文小节被外部引用时，在标题前加 `<a id="english-anchor"></a>`。
- `eqtools/csiExtend/docs` 可作为素材池和内部维护参考；公开用户路线以顶层 `docs/` 为唯一入口，不把内部设计、计划、审计和 changelog 原样搬进用户手册。
- `examples/` 只放短小任务例子，使用通用文件名和占位路径；完整真实事件脚本仍放在 `casebook/` 和 ECAT-Cases。
- `concepts/` 只解释跨页面复用的概念和对象关系，不放完整 API 字段表。

## 案例页面模板

```markdown
# 事件名：工作流类型

## 位置

## 为什么选这个案例

## 数据入口

## 运行方式

## 脚本对照

### 1. 第一步对应的代码

### 2. 第二步对应的配置

## 应保留输出

## 已知限制
```

## 短例页面模板

```markdown
# 任务名

一句话说明这个例子解决什么小任务。

## 输入

需要准备的最少文件、对象或配置。

## 最小代码

可复制代码或命令。文件名使用通用占位符，不写本地绝对路径。

## 检查

运行后应看到哪些输出，如何判断方向、单位、维度或数量是否合理。

## 何时不用这个例子

列出 2-3 个应转去 workflow、casebook 或 reference 的情况。

## 相关参考

链接到对应 workflow、reference 和案例。
```

## 概念页面模板

```markdown
# 概念名

一句话说明这个概念解决什么混淆。

## 核心判断

说明概念之间的关系、推荐路线或常见误区。

## 继续阅读

链接到 workflow、examples、reference 或 casebook。
```

## 整理技术材料时

从实现笔记、调试记录或案例脚本整理到公开手册时，要按用户学习路线重写：

- 从用户动作开始。
- 先给最小可运行命令。
- 当前阶段只解释必要参数。
- 复杂选项链接到 reference 页。
- 避免把未经整理的实验脚本直接写成教程。

## Reference 页面模板

`reference/` 页面可以很完整，但开头应保持轻量、可导航：

```markdown
# 页面标题

一句话说明本页负责什么，并链接到对应 workflow 或 casebook。

## 阅读路径

- 第一次运行应先看哪一页。
- 当前页中最常查的 2-4 个小节。
- 高级或维护场景应跳到哪里。

## 主体参考内容

字段、API、配置块、输出、诊断、常见误区和相关页面。
```

Reference 内容不要因为页面长而删除字段字典；应通过分组、阅读路径和相关页面链接降低首次阅读负担。
